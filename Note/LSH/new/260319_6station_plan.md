# 6개 스테이션 진행 방향 정리

## 대상 스테이션

```python
station_ids = [
    'ST-481',   # 상현
    'ST-2425',  # 다원
    'ST-1331',  # 찬솔
    'ST-454',   # 신영
    'ST-453',   # 혜전
    'ST-2264',  # 광태
]
```

## 기본 원칙

- 1차 타깃은 `total_flow`로 통일한다.
- 데이터 단위는 `1시간 1행`으로 맞춘다.
- `lag / rolling` 피처는 사용하지 않는다.
- 공통 피처는 `온도`, `습도`, `강수량`, `snow_flag`, `is_restingday`, `month_sin`, `month_cos`, `hour_sin`, `hour_cos`를 기본으로 둔다.
- 각 스테이션은 먼저 개별 패턴을 확인하고, 이후 6개 스테이션 통합 모델로 확장한다.

## 팀원별 스테이션 진행 방향

### 1. 상현 `ST-481`

- 현재 가장 많이 진행된 기준 스테이션으로 사용한다.
- 월별 시간대 패턴을 보고 군집화가 가능한지 먼저 확인한다.
- 대표 월 선택 실험, 단일 스테이션 예측, 외부 테스트까지 기준 파이프라인을 먼저 확정한다.
- 이후 다른 스테이션은 이 스테이션의 처리 흐름을 복사해서 맞춘다.

### 2. 다원 `ST-2425`

- 먼저 시간대별 `total_flow` 패턴이 `ST-481`과 비슷한지 확인한다.
- 단일 스테이션 기준으로 월별 패턴 안정성이 높은지 본다.
- 월별 변동이 크지 않으면 선형 계열과 트리 계열을 같이 비교한다.
- 변동이 크면 랜덤포레스트나 HGB 쪽을 우선 후보로 둔다.

### 3. 찬솔 `ST-1331`

- 출퇴근형인지 생활권형인지 시간대 피크를 먼저 본다.
- 평일/휴일 차이가 크면 `is_restingday` 효과를 집중적으로 본다.
- 월별 패턴이 일정하면 대표 월 선택 전략이 유효한지 확인한다.
- 월별 편차가 크면 전체 월 학습 방식이 더 적합한지 비교한다.

### 4. 신영 `ST-454`

- 유입/유출보다 먼저 `total_flow`의 안정성을 확인한다.
- 이상 시간대가 많으면 누락 시간 보정 이후 패턴이 유지되는지 본다.
- 군집 결과가 명확하면 대표 월 기반 실험을 적용한다.
- 군집이 약하면 시간순 train/valid/test 기본 분할로 간다.

### 5. 혜전 `ST-453`

- 다른 스테이션보다 계절성 차이가 큰지 먼저 확인한다.
- 월별 시간대 곡선이 비슷하면 군집 수를 작게 가져간다.
- 월별 차이가 크면 `month_sin`, `month_cos` 영향이 큰 모델을 우선 본다.
- 단일 스테이션 모델과 통합 모델에서 성능 차이가 큰지도 비교한다.

### 6. 광태 `ST-2264`

- 이용량 규모가 작으면 과적합 가능성을 먼저 본다.
- 이 경우 복잡한 모델보다 규제 선형모델과 랜덤포레스트를 우선 비교한다.
- 스테이션 단독으로 정보가 부족하면 6개 통합 모델에서 보완 가능한지 확인한다.
- 외부 테스트에서 성능 하락이 크면 통합 모델 의존도를 높인다.

## 공통 작업 순서

1. 원본 CSV 로드
2. `종료_대여소_ID == 'X'` 제거
3. `timestamp` 생성 또는 변환
4. 시간대별 `inflow`, `outflow`, `total_flow`, `net_flow` 집계
5. 누락 시간 보정
6. 월별 시간대 패턴 시각화
7. 월별 패턴 유사도 확인
8. 월별 군집화 여부 판단
9. 단일 스테이션 모델 학습
10. 외부 테스트 또는 연도 분리 테스트

## 6개 스테이션 통합 진행 방향

### 목표

- 스테이션별로 따로 모델을 만드는 방식과
- 6개 스테이션을 합쳐 하나의 모델을 만드는 방식을 비교한다.

### 통합 데이터 구성

- 각 스테이션을 모두 `1시간 1행`으로 맞춘 뒤 세로로 합친다.
- `station_id`를 명시적으로 유지한다.
- 스테이션 고정 정보가 있으면 같이 붙인다.
  - `위도`, `경도`
  - `residential_index`
  - `business_index`
  - `tourism_index`
  - 기타 입지 지표

### 통합 모델의 1차 타깃

- `total_flow`

이유:
- 유입/유출보다 안정적이다.
- 스테이션 간 비교가 더 쉽다.
- 1차 통합 모델 성능 확인에 적합하다.

### 통합 모델의 기본 피처

- 공통 시간/날씨 피처
  - `온도`, `습도`, `강수량`, `snow_flag`
  - `is_restingday`, `month_sin`, `month_cos`, `hour_sin`, `hour_cos`
- 스테이션 식별 피처
  - `station_id` 원-핫 또는 범주형 처리
- 스테이션 고정 피처
  - 입지/공간 특성 관련 수치형 컬럼

### 통합 모델에서 꼭 볼 비교

- 단일 스테이션 모델 vs 통합 모델
- 스테이션별 test 성능 차이
- 이용량이 작은 스테이션에서 통합 모델이 이득이 있는지

## 6개 스테이션 통합 검증 방향

### 방법 1. 시간 기준 분할

- 모든 스테이션에서 같은 날짜 구간으로 `train / valid / test`를 나눈다.
- 장점: 실제 운영 예측 상황과 가깝다.
- 단점: 스테이션별 데이터 편차를 직접 보기 어렵다.

### 방법 2. 스테이션별 점수 분리 확인

- 통합 모델을 학습한 뒤 `station_id`별로 test 성능을 따로 계산한다.
- 장점: 어떤 스테이션에서 통합 모델이 강한지 바로 확인 가능하다.

### 방법 3. Leave-one-station-out 검증

- 5개 스테이션으로 학습하고 1개 스테이션으로 테스트한다.
- 장점: 새로운 스테이션 일반화 가능성을 확인할 수 있다.
- 단점: 현재 목표가 운영 예측이면 다소 과한 검증일 수 있다.

## 추천 진행 순서

### 1단계

- 각 팀원이 자기 스테이션에서 `total_flow` 기준 단일 스테이션 모델을 만든다.
- 최소 비교 모델은 `RidgeCV`, `RandomForest`, `HGB`로 둔다.

### 2단계

- 각 스테이션의 최적 모델과 점수를 정리한다.
- 어떤 스테이션이 선형에 맞고, 어떤 스테이션이 비선형에 맞는지 비교한다.

### 3단계

- 6개 스테이션 통합 데이터셋을 만든다.
- `total_flow` 하나만 타깃으로 잡고 통합 모델을 먼저 만든다.

### 4단계

- 통합 모델과 단일 스테이션 모델을 비교한다.
- 특히 성능이 낮은 스테이션에서 통합 모델이 보완되는지 확인한다.

## 최종 정리 포인트

- 단일 스테이션 기준 최적 모델은 무엇인지
- 스테이션별로 패턴 차이가 얼마나 큰지
- 대표 월 전략이 실제로 도움이 되는지
- 6개 통합 모델이 개별 모델보다 나은 경우가 있는지
- 최종 운영용으로는 단일 모델이 나은지, 통합 모델이 나은지
